# Results Interpretation

This document explains how to read and interpret the results generated by this project.

---

## Understanding the Metrics

### Rolling Reward

The rolling reward is the average total episode reward over the last 100 episodes (default window). It smooths out episode-to-episode variance to show learning trends.

**How to interpret:**
- **Upward trend** → the agent is learning
- **Plateau** → the agent has converged (or stagnated)
- **Sudden drops** → the policy may have destabilized
- **High variance band** → the configuration is sensitive to random seed

**Caution:** Higher rolling reward does not necessarily mean better driving. The reward depends on the reward mode:
- R1 rewards survival only — a slow, cautious agent can score high
- R4 has higher reward magnitudes by design — R4 rewards are not directly comparable to R1 rewards

### Crash Rate

The percentage of episodes in which the agent goes off-track. This is a **safety metric**.

**How to interpret:**
- **Decreasing crash rate** → the agent is learning to stay on track
- **Persistently high crash rate** (>50%) → the reward signal is insufficient or the noise is too high
- **Very low crash rate** → could indicate cautious driving (check lap completion to confirm)

### Lap Completion Rate

The percentage of episodes in which the agent completes at least one full lap. This is a **task success metric**.

**How to interpret:**
- **Increasing lap rate** → the agent is learning to complete the racing task
- **Zero laps with high reward** → likely reward hacking (agent survives without racing)
- **High laps with moderate crash rate** → aggressive but effective driving

## Why High Reward Alone Can Be Misleading

Consider two hypothetical agents with R1 (basic) reward:

| Agent | Total Reward | Crashes | Laps | What Happened |
|-------|-------------|---------|------|---------------|
| Agent A | +15.0 | 0 | 0 | Sat still for 300 steps collecting +0.05/step |
| Agent B | +8.0 | 1 | 2 | Drove fast, completed 2 laps, then crashed |

Agent A has higher reward but Agent B is clearly the better driver. This is why **multiple metrics must be examined together**.

## Key Insights from Recent Results

### R3_N2 Performs Well

In recent experimental runs, **R3 (modified reward) with N2 (moderate noise, σ=0.02)** has shown strong overall performance:
- Good rolling reward
- Reasonable crash rate
- Solid lap completion

This suggests that the combination of informative reward shaping (R3's stability bonuses and speed scaling) with mild noise regularization (N2) produces robust policies.

### R1_N1 Can Show Misleading High Reward

R1_N1 (basic reward, no noise) sometimes shows surprisingly high rolling reward. This is because:
- The R1 reward is simple (+0.05 per step alive, −5.0 for crash)
- An agent that learns to drive slowly in safe areas maximizes cumulative survival reward
- No lap bonuses mean there is no incentive to actually race

Check crash rate and lap completion: if the agent has high reward but zero laps, it is reward hacking.

### Moderate Noise Helps Selected Reward Structures

N2 (σ=0.02) acts as a mild regularizer that can improve policy robustness:
- Works best with well-shaped rewards (R3, R2, R4) where the reward signal can guide learning through the noise
- Does not help with R1, where the reward signal is too sparse to compensate for perceptual uncertainty

### High Noise Generally Suppresses Learning

N3 (σ=0.05) tends to degrade performance across most configurations:
- The agent cannot reliably perceive track boundaries
- Q-value estimates become noisy and unreliable
- Crash rates increase as the agent cannot distinguish safe from dangerous states

## How to Read the Plots

### Individual Experiment Plots (`results/plots/{algo}/individual/`)

Each experiment folder contains three plots:
- `reward_vs_episodes.png` — Rolling reward curve with std band
- `crash_rate_vs_episodes.png` — Rolling crash rate
- `laps_vs_episodes.png` — Rolling lap completion

Look for:
- Overall trend direction
- Where the curve plateaus
- Width of the std band (stability across seeds)

### Comparison Plots (`results/plots/comparison/`)

These overlay TD3 and DDPG curves for the same configuration:
- Blue = TD3, Orange = DDPG
- Look for which algorithm converges faster, reaches higher reward, or has lower crash rate
- Check if one algorithm is consistently more stable (narrower std band)

### Grouped Plots (`results/grouped/`)

These show all four reward modes (R1–R4) at a fixed noise level for one algorithm:
- Useful for seeing how reward shaping affects performance under the same noise conditions
- Each curve is color-coded by reward level

## What Should Be Reported

In a research paper or report, present:

1. **Rolling reward curves** for key configurations (not all 24 — select representative ones)
2. **Summary table** with final-100-episode statistics: mean reward, crash rate, lap rate, for each configuration
3. **Trade-off analysis:** Highlight configurations where high reward coexists with high crash rate
4. **Noise interaction:** Show that N2 helps R3 but not R1 (non-monotonic finding)
5. **Algorithm comparison:** TD3 vs DDPG across metrics, noting TD3's stability advantage
6. **Error bars or std bands:** Always report across seeds, not from a single run
7. **Caveats:** Note that the current study uses 3 seeds per configuration; additional seeds would strengthen conclusions

---

## Interpreting Across Configurations

When comparing configurations, keep in mind:

- **Reward magnitudes differ between modes.** R4 generates higher absolute rewards than R1 by design. Compare within reward modes across noise levels, or normalize when comparing across modes.
- **Seed variability can be large.** A configuration that looks best in one seed may be mediocre in another. Always look at the mean and std.
- **No single winner.** Different configurations excel in different metrics. The "best" configuration depends on whether you prioritize reward, safety, or stability.
